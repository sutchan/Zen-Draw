01 / SEMANTIC

色彩语义

通过语义令牌引用颜色，而非直接使用 HEX 值 CSS Variables Theme-aware

每一种颜色都承担着明确的 UI 角色。使用语义化变量而非裸色值， 使得深色模式切换与未来的品牌调整成为一项替换令牌的工作。

02 / NEUTRAL

中性色阶

从 50 到 950 的十一级中性灰 · 构成所有界面的基底 11 Steps Monochrome

中性色阶是整个系统中使用最频繁的色彩。 浅色模式下低数值色号更接近白色，深色模式下低数值色号则作为"深底色"使用。

感知一致的中性灰

每一级色阶之间保持近似的感知亮度差。这意味着任何两级间的切换都会产生 可预测的视觉差异，可以被用于构建清晰的层级和对比。

03 / ACCENT

语义色

用于表达状态与反馈 · Success · Warning · Error · Info State Feedback

这些颜色传递明确的语义信号。它们应该被谨慎使用， 避免在同一界面上出现过多不同语义色，以免对用户造成认知负担。

色阶缩放

每一种语义色都有完整色阶，可以灵活应用于背景淡色、边框、主色与深色强调。

05 / PRINCIPLES

使用原则

保持一致、可预测与可访问的色彩使用方式

色彩的使用比色彩本身更重要。遵循以下原则，以确保不同团队、不同产品之间 视觉语言的一致性。

01

对比度要求

正文文本与背景的对比度不低于 4.5:1（AA 级）， 大号文本（18pt+）不低于 3:1。组件和交互元素保持一致的最小对比度。

  • 正文：4.5:1 (AA) / 7:1 (AAA)
  • 大号文本：3:1 (AA) / 4.5:1 (AAA)
  • UI 组件：3:1（非文字）
02

颜色分层

通过背景 → 表面 → 组件 → 强调的层次顺序构建界面。 每一层使用相对暗或亮的一档中性色来形成深度。

  • 背景：最深/最浅 — 作为画布基底
  • 表面：卡片、模态框 — 位于背景之上
  • 组件：按钮、输入 — 位于表面之上
  • 文字：与背景形成最高对比
03

避免仅用颜色

切勿仅通过颜色传达信息。必须同时提供图标、文字说明、 或形状的区别，以满足色盲用户与低视力用户的需求。

  • 状态：颜色 + 图标 + 文字
  • 标签：颜色 + 形状/符号
  • 图表：颜色 + 图案或标签
04

深色模式

深色模式不是简单的反色。降低饱和度、避免纯黑背景、 避免高对比度纯白，使用令牌驱动的主题切换而非硬编码颜色值。

  • 使用 #0a0a0a 而非 #000000 作为背景
  • 降低纯色色饱和度以避免光晕
  • 使用较浅的边框形成层次
  • 阴影可替换为发光边框效果
05

语义优先

使用语义化变量（--primary、--muted、--border）而不是具体色值（#f4f4f5）。 这使得主题切换、品牌更新变得容易维护。

  • 新增颜色需先定义其语义角色
  • 避免在业务代码中直接写 HEX
  • 通过 token 映射表追溯来源
06

克制使用

色彩应该被谨慎地、有目的地使用。在中性基底上使用少量的强调色 可以产生最大的视觉冲击力，避免"彩虹界面"。

  • 单个界面最多 1—2 种强调色
  • 语义色保留给状态反馈
  • 避免使用品牌色作为功能性色彩

A

色盲友好

红色与绿色在红绿色盲眼中高度相似。本系统通过 形状/图标/文字与颜色组合传递状态，而非仅依赖颜色差异。

B

焦点可见

所有交互元素聚焦时显示 3px --ring 环形轮廓， 与背景的对比度不低于 3:1，保障键盘用户的操作感知。

C

低亮度适配

深色模式下降低色彩饱和度避免"发光"效应， 并确保在 OLED 屏幕与低亮度环境下仍保持可读性。

D

减少闪烁

避免使用高频率闪烁的色彩动画，符合 WCAG 2.3 关于癫痫发作的规范。

07 / TOKENS

令牌映射

语义令牌 → CSS 变量 → Tailwind 类 → 实际值 Design Tokens

设计令牌是色彩系统的"源代码"。所有组件和业务代码都应该通过令牌引用颜色， 而不是直接使用 HEX 值。这使得主题切换、品牌调整变得简单而安全。
